Bootloader Architecture

How the Bootloader Works

When the microcontroller powers on or resets, it always starts by executing Stage 1 directly from the internal flash memory. Stage 1 is intentionally kept small and simple. Its primary job is to decide whether it's time to run the main application or if it needs to update the firmware instead.

To make this decision, Stage 1 checks two things: a special "Reboot Marker" in RAM (which the application can set when it wants to force an update) and the physical DFU hardware button. If neither of these are triggered, Stage 1 performs one final crucial test: it calculates a CRC32 checksum over the application flash memory. If the checksum proves the application is valid and uncorrupted, Stage 1 simply gets out of the way and jumps straight to the main application.

However, if an update is explicitly requested—or if the CRC check fails because the application is missing or corrupted—Stage 1 steps into action. Instead of running the updater directly from flash, it copies the Stage 2 bootloader payload from flash memory into RAM. Once the copy is finished, Stage 1 passes the baton and transfers execution over to Stage 2.

From this point on, Stage 2 runs entirely out of RAM. It initializes the USB interface and talks to your computer to securely download and write the new firmware to the flash memory. Once the flashing process is completely finished, Stage 2 clears the Reboot Marker and resets the microcontroller, allowing it to boot up normally into your brand new application.
